# 10. 后续原型与开发建议

## 1. 原型设计建议

建议先做低保真原型，再进入高保真 UI。

第一批页面：

1. 登录页
2. 创建账本页
3. 邀请成员页
4. 账本首页
5. 快速记账页
6. 流水详情页
7. 待结算页
8. 统计页
9. 设置页

原型重点：

- 记账是否足够快
- 分摊是否容易理解
- 待结算是否清楚
- 共同账单和个人账单是否容易区分
- 统计页是否能给出有价值的结论

## 2. UI 设计重点

参考用户截图，建议优先保留以下设计优点：

- 大圆角白色卡片
- 浅灰页面背景
- 大字号金额
- 底部悬浮导航
- 快速记账底部弹窗
- 设置页分组列表
- 分类图标网格
- 统计页模块化卡片

同时需要做家庭/情侣场景增强：

- 成员头像和成员名称需要频繁出现
- 共同账单、个人账单、私密账单要有明确标识
- 待结算金额要突出
- 分摊设置要简单可视化
- 统计页要给出自然语言结论

## 3. 技术架构建议

### 3.1 App 前端

可选方案：

| 方案 | 优点 | 适用情况 |
|---|---|---|
| Flutter | 双端一致性强，UI 还原度高 | 小团队快速做 iOS/Android |
| React Native | 适合已有 React 技术栈 | Web/前端团队转移动端 |
| 原生 Swift + Kotlin | 体验最好 | 预算充足、重体验项目 |

建议：如果团队较小，优先 Flutter。

### 3.2 后端

核心模块：

- 用户认证
- 账本管理
- 成员权限
- 流水管理
- 分摊结算计算
- 统计聚合
- 预算计算
- 文件上传
- 数据导入导出

### 3.3 数据库

建议使用 PostgreSQL 或 MySQL。

原因：

- 用户、账本、成员、流水、分摊关系明确
- 需要事务一致性
- 统计查询较多
- 权限关系较复杂

可搭配 Redis 做缓存：

- 首页统计缓存
- 月度统计缓存
- 用户会话
- 高频配置项

## 4. 同步与离线建议

记账 App 很适合做离线可用。

建议策略：

- 本地先保存账单
- 后台异步同步到云端
- 冲突时按更新时间和用户确认策略处理
- 同一账单编辑需要记录版本号

基础冲突规则：

```text
同一账单多人同时编辑
→ 检测版本号冲突
→ 提示用户选择保留本地/保留云端/手动合并
```

## 5. 权限与隐私建议

重要规则：

- 每个账本独立授权
- 私密账单仅本人可见
- 半私密账单隐藏备注和图片
- 导出数据需要二次验证
- 删除账本需要创建者确认
- 成员退出后历史账单仍保留，但可隐藏个人信息

## 6. 统计计算建议

建议做预聚合，提升首页和统计页加载速度。

预聚合维度：

- 月度收入
- 月度支出
- 月度结余
- 分类支出
- 成员付款
- 成员承担
- 预算使用
- 待结算金额

触发时机：

- 新增流水
- 编辑流水
- 删除流水
- 修改分摊
- 标记结算

## 7. 埋点建议

核心埋点：

- 注册完成
- 创建账本
- 邀请成员
- 成员加入
- 点击记账按钮
- 保存账单
- 使用分摊
- 查看待结算
- 标记已结算
- 查看统计页
- 设置预算
- 导入账单
- 导出账单

关键指标：

- 首次记账转化率
- 邀请成员转化率
- 分摊使用率
- 待结算完成率
- 7 日留存
- 30 日留存

## 8. 测试重点

### 功能测试

- 支出、收入、转账保存是否正确
- 编辑账单后金额是否正确更新
- 删除账单后统计是否回滚
- 账户余额是否正确变化

### 分摊测试

- 平均分金额是否准确
- 指定金额合计校验是否正确
- 付款人变化后待结算是否重算
- 多笔账合并结算是否准确

### 权限测试

- 成员是否只能查看授权账本
- 私密账单是否仅本人可见
- 访客是否无法新增或编辑账单
- 成员退出后权限是否失效

### 同步测试

- 离线记账后是否可同步
- 多设备编辑是否冲突提示
- 弱网下是否有明确反馈

## 10. 下一步建议

建议按照以下顺序继续推进：

```text
1. 确认 MVP 范围
2. 画低保真原型
3. 进行 5-10 组用户访谈
4. 调整核心流程
5. 输出高保真 UI
6. 输出接口文档和数据库 ER 图
7. 开始 V0.2 内测版开发
```

## 11. 最小开发闭环

第一版只要打通以下闭环，就可以内测：

```text
创建账本
→ 邀请成员
→ 记录共同支出
→ 自动分摊
→ 查看待结算
→ 标记已结算
→ 查看月度统计
```

这个闭环成立后，再逐步加入资产、预算、周期账单、导入导出等增强功能。
